üéØ Introduction to System Analysis & Design

What is System Analysis & Design?

System Analysis: Process of studying and analyzing existing systems to identify problems and requirements for new systems.

System Design: Process of defining components, modules, architecture, interfaces and data for a system to satisfy specified requirements.

Key Definitions (Must Remember!)

  • System: Collection of interrelated components working together toward common goal
  • Information System: Combination of hardware, software, data, network and people
  • Business Process: Set of activities performed to accomplish specific organizational goal
  • Stakeholder: Anyone who has interest in the system

System Analyst Roles

  • Problem solver and consultant
  • Bridge between business and technical teams
  • Requirements gatherer
  • Change agent

üîÑ System Development Life Cycle (SDLC)

SDLC Phases (Remember the sequence!)

  1. Planning: Define project scope, feasibility study
  2. Analysis: Gather requirements, study existing system
  3. Design: Create system architecture and detailed design
  4. Implementation: Code development and system building
  5. Testing: Verify system meets requirements
  6. Deployment: Install and make system operational
  7. Maintenance: Ongoing support and modifications

SDLC Models (Very Important for Exams!)

1. Waterfall Model

Advantages: Simple, easy to manage, well-documented

Disadvantages: No flexibility, late testing, no customer feedback

Use: When requirements are clear and stable

2. Iterative Model

Advantages: Early delivery, flexible, risk reduction

Disadvantages: More resources needed, complex management

Use: Large applications with clear requirements

3. Spiral Model

Advantages: Risk management, customer feedback, flexible

Disadvantages: Expensive, complex, requires expertise

Use: High-risk projects, large complex systems

4. Agile Model

Advantages: Customer collaboration, working software, responding to change

Disadvantages: Less documentation, requires skilled team

Use: When requirements change frequently

5. V-Model (Verification & Validation)

Advantages: Testing planned early, high success rate, good for small projects

Disadvantages: Rigid, expensive, no early prototypes

Use: When requirements are well understood and stable

6. Prototype Model

Advantages: User involvement, reduced risk, better requirements understanding

Disadvantages: Time consuming, may lead to poor design

Use: When requirements are unclear or when user interface is critical

7. RAD Model (Rapid Application Development)

Phases: Planning ‚Üí User Design ‚Üí Construction ‚Üí Cutover

Advantages: Quick development, user feedback, reusable components

Disadvantages: Requires skilled developers, not suitable for large projects

Use: When system can be modularized and time is constraint

8. Incremental Model

Method: System developed and delivered in increments

Advantages: Early delivery, easier testing, flexible

Disadvantages: Requires good planning, integration issues

Use: When partial implementation is acceptable

üîç System Analysis

Requirements Gathering Techniques

  • Interviews: Face-to-face discussion with stakeholders
  • Questionnaires: Written questions for large groups
  • Observation: Watching current system in operation
  • Document Analysis: Studying existing documentation
  • Prototyping: Building preliminary version

Types of Requirements

  • Functional Requirements: What system should do
  • Non-functional Requirements: How system should perform (performance, security, usability)
  • Business Requirements: High-level business needs
  • User Requirements: What users need to accomplish

Feasibility Study (Very Important!)

Types of Feasibility:

  • Technical Feasibility: Can it be built with available technology?
  • Economic Feasibility: Is it cost-effective? (Cost-Benefit Analysis)
  • Operational Feasibility: Will it work in current environment?
  • Schedule Feasibility: Can it be completed on time?
  • Legal Feasibility: Any legal issues?

üé® System Design

Design Principles

  • Modularity: Divide system into manageable modules
  • Abstraction: Hide complex implementation details
  • Cohesion: Elements within module work together
  • Coupling: Minimize dependencies between modules
  • Information Hiding: Encapsulate implementation details

Types of Design

Logical Design:

What the system will do - independent of hardware/software

  • Data flow diagrams
  • Entity relationship diagrams
  • Process specifications

Physical Design:

How the system will be implemented

  • Hardware specifications
  • Software architecture
  • Database design
  • User interface design

Database Design Concepts

  • Normalization: Eliminate data redundancy (1NF, 2NF, 3NF)
  • Primary Key: Unique identifier for each record
  • Foreign Key: Reference to primary key in another table
  • Referential Integrity: Maintain consistency between related tables

üõ†Ô∏è Tools & Techniques

Data Flow Diagram (DFD) - Very Important!

DFD Components:

  • Process (Circle): Transform input to output
  • Data Store (Open Rectangle): Repository of data
  • External Entity (Square): Outside system boundary
  • Data Flow (Arrow): Movement of data

DFD Levels:

  • Context Diagram (Level 0): Highest level, shows system boundary
  • Level 1: Major processes of the system
  • Level 2+: Detailed breakdown of processes

Entity Relationship Diagram (ERD)

ERD Components:

  • Entity (Rectangle): Object or concept
  • Attribute (Oval): Property of entity
  • Relationship (Diamond): Association between entities
  • Cardinality: One-to-One, One-to-Many, Many-to-Many

Other Important Tools

  • Use Case Diagram: Shows interactions between users and system
  • Activity Diagram: Shows workflow and business processes
  • State Diagram: Shows different states of system
  • Sequence Diagram: Shows interaction over time
  • Class Diagram: Shows structure of system classes

‚öôÔ∏è System Implementation

Implementation Strategies

1. Direct Conversion

Method: Immediately replace old system with new

Advantage: Quick, cost-effective

Disadvantage: High risk, no backup

2. Parallel Conversion

Method: Run both systems simultaneously

Advantage: Low risk, backup available

Disadvantage: Expensive, resource intensive

3. Phased Conversion

Method: Implement system in phases

Advantage: Manageable risk, gradual learning

Disadvantage: Longer implementation time

4. Pilot Conversion

Method: Implement in one department first

Advantage: Test before full implementation

Disadvantage: May not represent full system

Change Management

  • User Training: Prepare users for new system
  • Communication: Keep stakeholders informed
  • Support: Provide ongoing assistance
  • Documentation: User manuals and technical docs

üß™ System Testing

Types of Testing (Remember for Exams!)

  • Unit Testing: Test individual components
  • Integration Testing: Test component interactions
  • System Testing: Test complete system
  • Acceptance Testing: User validates system meets requirements
  • Performance Testing: Test system under load
  • Security Testing: Test system vulnerabilities

Testing Strategies

Black Box Testing:

Test functionality without knowing internal structure

Techniques: Equivalence partitioning, boundary value analysis

White Box Testing:

Test internal structure and logic

Techniques: Statement coverage, path testing

Quality Assurance

  • Verification: Are we building the product right?
  • Validation: Are we building the right product?
  • Quality Control: Process of testing and reviewing
  • Quality Metrics: Measures of system quality

üîß Advanced Development Methods & Techniques

Structured Analysis Methods

1. Gane & Sarson Method

Focus: Data Flow Diagram notation and structured analysis

Key Features:

  • Rounded rectangle processes
  • Open rectangles for data stores
  • Squares for external entities
  • Curved arrows for data flows

Use: Traditional structured analysis projects

2. Yourdon Method

Focus: Complete structured methodology

Components:

  • Circles for processes in DFD
  • Parallel lines for data stores
  • Boxes for terminators
  • Arrows for data flows

Phases: Survey ‚Üí Analysis ‚Üí Design ‚Üí Implementation

3. DeMarco Method

Emphasis: Specification-driven development

Key Concepts:

  • Data dictionary usage
  • Process specifications
  • Leveled data flow diagrams
  • Structured English

Object-Oriented Methods

1. OMT (Object Modeling Technique)

Three Models:

  • Object Model: Static structure (classes, objects)
  • Dynamic Model: Temporal behavior (state diagrams)
  • Functional Model: Data transformations (DFDs)

Phases: Analysis ‚Üí System Design ‚Üí Object Design ‚Üí Implementation

2. Booch Method

Focus: Object-oriented design and architecture

Key Features:

  • Class diagrams
  • Object diagrams
  • Interaction diagrams
  • Module diagrams

3. Jacobson Method (OOSE)

Object-Oriented Software Engineering

Core Concepts:

  • Use cases for requirements
  • Object interaction diagrams
  • Activity diagrams
  • Component architecture

Modern Agile Methodologies

1. Scrum Framework

Roles: Product Owner, Scrum Master, Development Team

Events: Sprint, Sprint Planning, Daily Scrum, Sprint Review, Retrospective

Artifacts: Product Backlog, Sprint Backlog, Increment

Sprint Duration: 1-4 weeks (typically 2 weeks)

2. Extreme Programming (XP)

Core Practices:

  • Pair Programming
  • Test-Driven Development (TDD)
  • Continuous Integration
  • Refactoring
  • Small Releases

Values: Communication, Simplicity, Feedback, Courage

3. Kanban Method

Principles:

  • Visualize workflow
  • Limit work in progress (WIP)
  • Measure and manage flow
  • Make process policies explicit

Board Columns: To Do ‚Üí In Progress ‚Üí Testing ‚Üí Done

4. Feature Driven Development (FDD)

Five Processes:

  1. Develop overall model
  2. Build features list
  3. Plan by feature
  4. Design by feature
  5. Build by feature

Hybrid & Specialized Methods

1. Unified Process (UP)

Phases: Inception ‚Üí Elaboration ‚Üí Construction ‚Üí Transition

Disciplines: Requirements, Analysis, Design, Implementation, Test

Characteristics: Iterative, incremental, use-case driven, architecture-centric

2. Crystal Methods

Colors by Team Size:

  • Crystal Clear (2-6 people)
  • Crystal Yellow (7-20 people)
  • Crystal Orange (21-40 people)
  • Crystal Red (41-80 people)

3. Dynamic Systems Development Method (DSDM)

Principles:

  • Focus on business need
  • Deliver on time
  • Collaborate
  • Never compromise quality
  • Build incrementally

Analysis Techniques

1. Joint Application Development (JAD)

Purpose: Gather requirements through structured workshops

Participants: Users, developers, facilitator, scribe

Benefits: Faster requirements gathering, better user buy-in

2. Business Process Reengineering (BPR)

Definition: Fundamental rethinking and radical redesign of business processes

Goals: Dramatic improvements in cost, quality, service, speed

Steps: Map current process ‚Üí Analyze ‚Üí Redesign ‚Üí Implement

3. Rich Picture Technique

Purpose: Visual representation of problem situation

Elements: Stakeholders, processes, concerns, conflicts, relationships

Use: Problem understanding and communication

Software Metrics & Estimation

Function Point Analysis (FPA)

Purpose: Measure software size based on functionality

FP = (EI √ó 4) + (EO √ó 5) + (EQ √ó 4) + (ILF √ó 10) + (EIF √ó 7)
  • EI: External Inputs
  • EO: External Outputs
  • EQ: External Queries
  • ILF: Internal Logical Files
  • EIF: External Interface Files

COCOMO Model (Cost Estimation)

Basic COCOMO Formula:

Effort = a √ó (KLOC)^b Time = c √ó (Effort)^d

Project Types:

  • Organic: Small teams, familiar environment
  • Semi-detached: Medium teams, mixed experience
  • Embedded: Large teams, complex constraints

Software Quality Metrics

  • Cyclomatic Complexity: V(G) = E - N + 2P
  • Defect Density: Defects per KLOC
  • Code Coverage: % of code tested
  • Mean Time Between Failures (MTBF)
  • Customer Satisfaction Index

Risk Management in SAD

Risk Categories

  • Technical Risks: Technology, complexity, performance
  • Project Risks: Schedule, resources, scope creep
  • Business Risks: Market, financial, competitive
  • External Risks: Regulatory, environmental, supplier

Risk Management Process

  1. Risk Identification: Brainstorming, checklists, interviews
  2. Risk Analysis: Probability √ó Impact assessment
  3. Risk Planning: Mitigation strategies
  4. Risk Monitoring: Track and review
Risk Exposure = Probability √ó Impact

Risk Mitigation Strategies

  • Avoid: Eliminate the risk source
  • Transfer: Shift risk to third party
  • Mitigate: Reduce probability or impact
  • Accept: Acknowledge and monitor

Configuration Management

Version Control Concepts

  • Baseline: Approved configuration item
  • Branch: Parallel development line
  • Merge: Combine changes from branches
  • Tag: Label for specific version
  • Repository: Central storage for all versions

Change Control Process

  1. Change Request: Formal request submission
  2. Impact Analysis: Assess effects and costs
  3. Change Approval: Decision by change board
  4. Implementation: Make the changes
  5. Verification: Ensure changes work correctly

System Security & Controls

Security Requirements

  • Authentication: Verify user identity
  • Authorization: Control access to resources
  • Confidentiality: Protect sensitive data
  • Integrity: Ensure data accuracy
  • Availability: System accessibility
  • Non-repudiation: Prevent denial of actions

Access Control Models

  • DAC (Discretionary): Owner controls access
  • MAC (Mandatory): System enforces access rules
  • RBAC (Role-Based): Access based on roles
  • ABAC (Attribute-Based): Dynamic access decisions

Security Testing Types

  • Vulnerability Assessment: Find security weaknesses
  • Penetration Testing: Simulate attacks
  • Security Auditing: Review compliance
  • Code Review: Static analysis for vulnerabilities

Performance Engineering

Performance Metrics

  • Response Time: Time to complete request
  • Throughput: Requests processed per unit time
  • Latency: Delay in processing
  • Scalability: Ability to handle increased load
  • Resource Utilization: CPU, memory, disk usage

Performance Testing Types

  • Load Testing: Normal expected load
  • Stress Testing: Beyond normal capacity
  • Volume Testing: Large amounts of data
  • Spike Testing: Sudden load increases
  • Endurance Testing: Extended periods

Maintenance & Evolution

Types of Maintenance

  • Corrective: Fix bugs and errors
  • Adaptive: Adapt to environment changes
  • Perfective: Improve performance/functionality
  • Preventive: Prevent future problems

Legacy System Evolution

  • Wrapping: Add new interface to old system
  • Reengineering: Restructure existing system
  • Migration: Move to new platform
  • Replacement: Build completely new system
